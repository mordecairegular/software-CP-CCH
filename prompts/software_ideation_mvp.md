# Step 2A 无软件时的 MVP 定义与开发门槛

## 目标

当用户尚无软件时，先把方向收敛成一个真实、可运行、可解释、可截图、可提交材料的最小软件。不要一上来写软著文档。

## 原则

- 软件应解决一个具体使用场景，不做空泛“平台”。
- 代码必须能运行，核心功能必须可操作、可截图、可说明。
- 不为凑行数生成无意义类、重复页面或无业务价值的样板代码。
- 如使用 AI 生成代码，必须由用户/agent 完成需求定义、选择、修改、整合和测试，并保留记录。

## 需求收敛

向用户确认或根据上下文推断：

1. 目标用户是谁？
2. 软件处理什么对象？例如项目、设备、报告、合同、数据集、客户、库存、工单。
3. 三个核心功能是什么？
4. 输入、处理、输出分别是什么？
5. 是否需要登录、权限、导出、报表、日志、配置、数据持久化？
6. 运行方式是 Web、桌面、小程序、脚本工具还是服务端 API？

## 推荐 MVP 构成

一个可申请的最低软件通常至少包含：

- 入口与导航。
- 3 个以上真实功能模块，或 1 个完整业务闭环加辅助模块。
- 数据录入、列表/状态展示、处理逻辑、结果输出或导出。
- README、运行命令、版本号、主要依赖说明。
- 基础测试、样例数据或演示脚本。
- 可截图的操作状态。

## 开发质量检查

开发完成前不得进入材料生成，除非满足：

- 能从干净环境按 README 启动。
- 关键功能至少手动跑通一次。
- 代码有清晰模块边界，不是一页脚本堆全部逻辑。
- 没有大量 `TODO`、占位按钮、假数据冒充真实处理结果。
- 说明书中的每个功能都能在软件中找到对应界面、命令或 API。
- 源码中不存在密钥、个人隐私、未授权第三方代码或明显侵权素材。

## 交付记录

在 `软件著作权申请资料/草稿/MVP开发记录.md` 记录：

- 需求定义时间和版本。
- 人工决策点。
- AI 参与范围。
- 关键修改与测试结果。
- 可运行命令和截图计划。
