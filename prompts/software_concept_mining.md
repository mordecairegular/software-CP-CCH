# Step 2 软著候选挖掘与灵魂审查

## 目标

在决定“申请哪个软件”或“先开发哪个软件”之前，先把候选软件从项目资料、业务场景和申报需求中挖深。不要只按文件名、技术栈或现有小脚本包装成软著。

本步骤回答三个核心问题：

1. 这个软件真正解决什么工程/业务问题？
2. 它对实际工作有什么指导意义或执行价值？
3. 谁在什么地方、什么时间、用它完成什么任务，并产出什么可交付证据？

## 必须产出

生成 `analysis/候选软件挖掘.json` 和 `草稿/软著候选挖掘.md`，至少包含：

- 候选软件清单：不少于 3 个候选方向，除非项目证据只支持 1 个方向。
- 每个候选的软件名称建议、版本建议、适用用户、业务对象、应用场景、使用时机。
- “灵魂问题”：该软件解决的痛点不是“提高效率”这类空话，而是某个具体任务中的数据错误、流程返工、资料不一致、计算难复核、报告难归档、申报资料不合规等问题。
- 工程实践联系：输入数据、校验规则、处理流程、输出结果、异常处理、日志/留痕、复核/导出如何指导真实业务。
- 申报联系：该软件如何支撑项目申报、资料审查、验收归档、内审复核、合同/设计/运维交付中的某个环节。
- 可申请性判断：源码证据、界面证据、截图证据、操作说明书可写性、源程序鉴别材料支撑度。
- 深度不足补强方案：如果现有成果太浅，列出必须开发的真实功能、数据结构、界面和输出材料。
- 风险：权属、AI 辅助、开源依赖、样例数据、夸大功能、跨项目残留。

`analysis/候选软件挖掘.json` 至少包含：

- `candidates[]`
- `candidate_id`
- `recommended`
- `score`
- `evidence_files[]`
- `source_boundary[]`
- `screenshots_needed[]`
- `risks[]`
- `decision_required`

## 候选评分表

每个候选按 0-5 分打分：

| 维度 | 说明 |
|------|------|
| 业务痛点清晰度 | 是否能说清真实使用者和任务场景 |
| 工程实践价值 | 是否能指导检查、计算、转换、复核、归档或决策 |
| 软件表达完整度 | 是否有数据模型、界面、处理逻辑、输出和异常处理 |
| 申报材料支撑度 | 是否容易生成申请表、源码材料、说明书和截图 |
| UI/操作证据质量 | 是否有完整、可信、专业的界面和操作流程 |
| 权属与真实性风险 | 风险越低分越高 |

总分低于 20 的候选不得直接进入正式材料；应先补强或放弃。

## 灵魂审查问题

对推荐候选必须逐条回答：

- 用户是谁：岗位、角色、专业背景。
- 用在哪里：办公室、设计院、项目公司、施工/运维现场、资料报送前、内部审查会等。
- 什么时候用：资料收集、格式转换、申报前质检、批量复核、成果归档、问题整改、提交代理前。
- 输入是什么：文件、表格、坐标、文本、图片、报告、配置项、业务参数。
- 软件做什么：校验、转换、计算、比对、生成、汇总、留痕、导出。
- 输出是什么：质检表、申报包、日志、报告、图件、源文件副本、异常清单。
- 业务后果是什么：减少返工、发现资料缺项、统一格式、保留审查证据、支持人工复核。
- 不能做什么：不得写不存在的自动审批、智能决策、云端协同、行业平台能力。

## 决策规则

- 不要在第一版候选未经过用户或上下文确认时匆忙开发。
- 如果用户要求直接执行，可以先做候选矩阵，然后选择得分最高且证据最扎实的候选进入开发，但必须在交付说明中记录选择理由。
- 如果现有项目只是脚本或简陋 demo，应优先补成“真实业务闭环 + 可视化操作界面 + 输出证据包”，而不是靠文案包装。
- 候选名称要贴近软件实际功能和业务对象，避免“大平台、智能管理系统、全流程系统”等超出源码证据的包装。

确认推荐候选后运行：

```bash
python tools/confirm_stage.py --workdir 软件著作权申请资料 --stage candidate --note "<用户确认的 candidate_id、软件名称和申报边界>"
```
