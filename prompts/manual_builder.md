# Step 7 操作手册 / 软件说明书

## 目标

生成面向软著审核和普通读者的操作手册或软件说明书草稿，并为最终 Word 操作说明书准备章节、截图和图题。重点说明软件用途、功能、界面、操作流程和结果反馈。

## 结构建议

按项目实际组织，不强制固定模板。常见章节：

1. 软件概述。
2. 运行环境。
3. 软件启动与登录/入口。
4. 主界面与导航说明。
5. 核心功能模块操作说明。
6. 典型业务流程。
7. 输出结果、报表或数据字段说明。
8. 使用注意事项和常见问题。
9. 技术特点简述。

## 截图规则

- 优先使用真实运行截图。
- 每个核心模块至少有一个截图，重要模块建议初始状态和结果状态各一张。
- 截图应展示完整窗口或完整页面关键区域，不要只截按钮。
- 暂无截图时保留正式 Word 可见占位，例如 `【截图预留：项目列表页面】`，不要用隐藏注释。
- 草稿中可以使用 Mermaid 等代码描述流程图，但正式 Word 必须转换为图片插入，不能把代码块原文放进说明书正文。

截图方式必须让用户选择：

- 浏览器自动截图。
- 桌面自动截图。
- 用户自行提供截图。
- 暂时跳过并保留占位。

确认后运行：

```bash
python tools/confirm_stage.py --workdir 软件著作权申请资料 --stage screenshot-method --method <browser|desktop|user-supplied|skip> --note "<用户选择>"
```

## 去低质 AI 味

禁止：

- 每节同一种句式。
- 大量“赋能、智能化、高效便捷、显著提升、全方位、一站式”。
- 没有页面对象、操作动作、系统反馈的空泛正确话。
- 写不存在的功能。
- 把框架、接口、状态管理细节写成用户操作。

每个模块至少回答：

- 这个模块处理什么对象？
- 用户在哪里看到它？
- 用户执行什么操作？
- 软件给出什么结果或反馈？
- 对应证据是什么？

## 输出

先生成 `草稿/操作手册.md` 和 `草稿/操作手册自检记录.md`。自检至少三轮：

1. 章节完整性和截图位置。
2. 项目证据回溯。
3. 去制式表达和 AI 味。

正式交付阶段必须继续进入 `word_final_builder.md`，生成 `03 <软件全称><版本号> 操作说明书.docx`。说明书 Word 应插入真实截图或醒目占位，并检查图片不溢出、章节与截图对应、软件名版本号一致。

正式说明书不得包含草稿、过程记录、原始 Markdown 标题、Mermaid 代码块或“原始操作手册草稿摘要”。这些内容只能留在过程文件或 QA 记录中。
