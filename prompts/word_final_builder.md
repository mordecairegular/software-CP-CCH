# Word 最终交付生成流程

## 目标

把已确认或待确认的软著材料落到可交给代理机构复核的 Word 文件，而不是停留在 Markdown 草稿。默认生成三份文件：

1. `01 计算机软件著作权登记申请表 — <软件全称><版本号>.docx`
2. `02 软件源代码 — <软件简称或英文名> <版本号>.docx`
3. `03 <软件全称><版本号> 操作说明书.docx`

## 前置条件

- 已完成业务理解、申请表字段、代码选择、操作手册和截图草稿。
- 源码必须来自真实项目或本轮实际开发项目。
- 若著作权人、开发完成日期、发表状态等字段尚未确认，Word 文件只能标为“待确认版”，不得声称可直接提交。
- 若使用申请表模板，优先使用用户提供或代理确认过的 `.docx` 模板，并通过 `--template-dir` 传入；不要把无关页眉页脚当成退回风险，重点保证内容和对应格式正确。

## 生成要求

### 申请表

- 基于模板或等价表单结构生成 `.docx`。
- 不得只按裸坐标填单元格；必须识别合并单元格后的唯一单元格，并按“软件名称、版本号、著作权人、开发完成日期、主要功能”等标签定位。
- 保存后重新打开 Word 文件，回读关键字段，确认软件名、版本号、著作权人、源程序量、主要功能没有落错位置。
- 对待确认字段使用醒目的 `【待确认：...】`，并在生成报告中列出。

### 源程序鉴别材料

- 源码行必须来自真实文件。
- 源码总量少于官方/代理要求的阈值时，按全量提交处理，并在报告中说明真实行数；不得为凑页数编造代码。
- 源码总量超过阈值时，按一般交存规则抽取前后连续页/行，保持文件顺序和连续性。
- 使用等宽字体；每页目标不少于 50 行代码或按代理模板执行。模块说明可以帮助阅读，但不得冒充源码行数。

### 操作说明书

- 说明书应包含封面、软件简介、运行环境、启动方式、主界面、功能模块、输出结果、计算方法/技术特点、常见问题或注意事项。
- 每个核心功能模块至少有初始/配置状态和运行/结果状态截图；代理有特别意见时，以“7 张以上完整窗口截图、15 页以上说明书”为优先目标。
- 截图应来自真实软件界面或真实运行页面，尽量裁去浏览器/IDE外框；如只能提供浏览器页面截图，需要在报告中说明。
- 说明书、申请表和源码材料中的软件全称、简称、版本号必须一致。

## 推荐工具

可运行：

```bash
python tools/word_material_builder.py --workdir <软件著作权申请资料目录>
```

常用可选参数：

```bash
python tools/word_material_builder.py \
  --workdir <目录> \
  --template-dir <代理样本文档目录> \
  --owner "<著作权人>" \
  --developer "<开发者>" \
  --completion-date "2026年06月09日"
```

如果没有模板目录，工具会生成结构化申请表草稿，但应提示用户或代理再按当前官方/代理模板复核。

生成后必须查看：

- `正式资料/word/word_generation_report.md`
- 三份 `.docx`
- 如使用 Documents skill，还要渲染为 PNG 或 PDF 做视觉 QA。

## 自检问题

- 三份 Word 的软件名称、版本号是否完全一致？
- 申请表每个关键字段是否写入正确标签旁边？
- 是否仍有 `待确认` 字段？
- 源码材料是否来自真实源码，且行数/页数口径说明清楚？
- 说明书截图是否足够、完整、清晰，并覆盖主要功能？
- Word 文件是否能打开、表格不乱、图片不溢出、页面没有明显空白或文字遮挡？
