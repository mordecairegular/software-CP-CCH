# Step 1 边界与输入

## 目标

用少量问题判断用户处于哪条路线：已有软件申请、先开发再申请、补正/迭代。

## 开场

```
先确认软著案件边界。软著重点是软件表达、真实源码、文档和权属一致；不是专利查新。信息不全可以先跳过，我会把假设写进草稿。
```

## 必问问题

1. 当前是否已经有可运行软件？
   - 有：提供项目路径、仓库、压缩包、README、截图或运行方式。
   - 没有：用一句话说想做的软件方向、使用者和核心场景。
   - 已有材料/补正：提供上一版材料或补正意见。
2. 申请主体是谁？
   - 自然人、公司、学校/事业单位、合作开发、委托开发、职务开发、受让/继承。
3. 软件名称与版本是否已定？
   - 如未定，先用候选名，后续必须确认。
4. 本轮申报目标是否已定？
   - 单一软件 / 多个软件之一 / 多个软件分别申报 / 尚未确定。
   - 如尚未确定，必须先做候选软件挖掘和申报对象确认。
5. 实际使用单位/岗位、业务场景和预期登记名称是什么？
   - 如果用户暂时说不清，记录为待确认，不得用空泛包装替代。
6. 是否使用 AI 辅助开发、开源项目、第三方素材、模型 API 或外包人员？
   - 只需如实说明，后续用于权属和材料口径审计。
7. 是否需要本轮同时开发一个最低可申请软件？

## 路由规则

- 有真实项目路径或源码：进入 `project_scan.md`，再进入 `software_concept_mining.md` 确认申报对象。
- 没有软件但想申请：进入 `software_concept_mining.md`，比较 2-3 个可申请 MVP 候选后，再进入 `software_ideation_mvp.md`。
- 有补正意见、代理反馈或上一版材料：进入 `iteration_context.md`，只处理差异点。
- 权属明显不清：进入 `rights_ai_audit.md`，先澄清，不要继续生成正式材料。

## 汇总输出

用 5 行以内复述：

- 路线：已有软件 / 先开发 / 补正迭代
- 项目或方向
- 申请主体
- AI/开源/第三方参与情况
- 申报对象是否已确认
- 下一步与需要用户确认的点
