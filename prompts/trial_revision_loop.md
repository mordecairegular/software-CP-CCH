# 试用修订沉淀流程

## 目标

用户试用本 skill/agent 的过程本身就是成型材料。每一轮反馈都要沉淀为可追溯记录，避免“当场修了，但后来忘了为什么修”。

## 先分类

收到用户反馈后，先判断属于哪一类：

- **案件材料修订**：修改某个软著案件的申请表、手册、源码材料、截图、权属记录。写入该案件目录的 `草稿/修订记录.md`。
- **skill/agent 修订**：修改本 skill 的流程、提示词、脚本、质量规则、安装方式、命名、输出体验。写入 skill 仓库的 `docs/trial-revision-log.md`，必要时再改文件。
- **双重影响**：既要修案件材料，又暴露了流程缺陷。两边都记录。

## 记录先于修改

凡是 skill/agent 自身修订，先追加记录，再编辑文件。记录至少包含：

- 时间。
- 试用场景或案件阶段。
- 用户原始反馈摘要。
- 观察到的问题。
- 影响范围：入口、流程、prompt、reference、tool、README、安装。
- 修订决策。
- 改动文件。
- 验证方式和结果。
- 需要以后回归测试的场景。
- 未决问题。

可运行：

```bash
python tools/revision_log.py --summary "<反馈摘要>" --decision "<修订决策>" --files "<改动文件，逗号分隔>" --verify "<验证方式/结果>"
```

## 修改后复核

修订完成后：

1. 简短复述这次反馈如何改变了 skill。
2. 如改动脚本，运行最小验证。
3. 如改动流程或 prompt，检查 `SKILL.md` 是否仍然短而清楚。
4. 如改动安装包，重新同步到安装目录。

## 不要做的事

- 不把用户一句临时偏好变成全局规则，除非它可复用。
- 不把案件事实写进 skill 通用规则。
- 不用“已优化”糊弄过去，必须说明具体改了什么。
- 不覆盖历史试用记录。
