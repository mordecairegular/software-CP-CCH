# Step 2B 项目扫描

## 目标

读取已有项目，建立材料生成的证据基础：源码、文档、运行方式、功能入口、截图、版本号、依赖和风险线索。

## 扫描步骤

1. 列出项目根目录，优先使用 `rg --files`。
2. 读取 README、package/build 配置、入口文件、路由、页面、服务、模型、核心业务逻辑。
3. 查找现有文档：PRD、设计说明、用户手册、需求文档、架构图、截图。
4. 识别运行方式和是否能启动。
5. 统计源码文件、源码行数、主要语言、候选抽取文件。
6. 识别可能独立申报的软件边界：前端应用、后台服务、命令行工具、数据处理模块、可视化/报告模块、管理后台或业务子系统。
7. 识别 UI 技术栈、组件库、截图证据和视觉风险；不要只判断“有窗口”。
8. 搜索风险词：密钥、个人信息、未完成占位、测试账号、AI 生成标记、第三方版权声明。

## 可用脚本

```bash
python tools/project_audit.py --project <项目路径> --out-dir 软件著作权申请资料/analysis
```

脚本只收集证据，不能替代模型判断。模型必须阅读输出的 `project_audit.md/json` 和必要源码，再进入 `software_concept_mining.md`，形成申报对象候选。

## 输出

生成或整理：

- `analysis/project_audit.md/json`
- `analysis/候选软件挖掘.json` 和 `草稿/软著候选挖掘.md` 的证据输入
- 软件名称候选
- 版本号候选
- 运行命令候选
- 核心功能证据
- 独立申报候选边界
- 代码抽取候选
- 截图候选和缺口
- UI 技术栈、组件库和设计风险
- 风险线索

## 停止条件

以下情况要先停下，不要直接生成材料：

- 项目无法定位或有多个候选项目。
- 仓库中存在多个可独立申报的软件候选，但尚未比较和确认申报对象。
- 看不到真实源码。
- 软件名称、版本号或申请主体完全不明。
- 代码明显只是空壳或演示模板，无法支撑说明书。
- 发现密钥、个人隐私或明显第三方侵权素材。
