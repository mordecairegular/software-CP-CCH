# 外部项目调研摘记

## GitHub

### Fokkyp/SoftwareCopyright-Skill

链接：https://github.com/Fokkyp/SoftwareCopyright-Skill

可借鉴：

- 将脚本限定为证据收集、校验和文件生成，行业判断与代码选择由 agent 阅读项目后完成。
- 先业务理解，再申请表、手册和代码材料。
- 代码材料必须来自真实源码，先生成候选清单，再由模型选择并让用户确认。
- 关键节点设置人工门禁。
- 草稿和正式资料分目录存放，生成报告留档。

不足：

- 主要面向已有项目，不覆盖“先确定方向并开发最低可申请软件”的完整路径。
- 对 AI 辅助开发、权属审计、开源许可证和低质 AI 代码风险可进一步强化。

### suzuka61/RuanZhu

链接：https://github.com/suzuka61/RuanZhu

可借鉴：

- 文档导入、信息提取、用户确认、流式生成、图片嵌入和目录生成体验。
- 软件类型模板可作为缺失字段提示，但不能替代项目证据。
- 去 AI 味意识值得保留。

风险：

- 模板填充容易把不存在的功能或技术架构写进材料。
- 主要生成概述和设计说明书，不是完整材料证据链。

### dts-gxu/copyright-generator

链接：https://github.com/dts-gxu/copyright-generator

可借鉴：

- 将生成拆为代码、说明书、截图、申请表和打包输出流水线。
- 截图和文档裁剪工具思路可参考。

风险：

- 直接生成代码用于申请是高风险路径。若用户没有软件，应先开发真实 MVP、运行验证并留开发记录，而不是生成一次性材料包。

## Hugging Face / GPTs

公开检索没有发现直接适配中国软著申请全流程的高质量 Hugging Face 项目。可借鉴点主要是模型生态和中文语料/行业表达，但不能用外部模型或数据集编造当前项目功能。

GPTs 类工具多偏提示词和文档生成，适合作为交互体验参考，不适合作为真实性、权属和代码证据链的依据。
