# 软著申请全流程 Agent PRD

## 目标

构建一个面向中国计算机软件著作权登记的全流程 skill。它既能处理已有软件的材料整理，也能在用户只有方向时，引导定义并开发一个真实、可运行、可申请的最小软件。

## 非目标

- 不替代中国版权保护中心线上填报系统。
- 不输出法律意见或保证登记通过。
- 不编造源码、截图、开发记录、权属证明或第三方授权。
- 不为了页数制造无业务意义代码。

## 核心用户流程

```text
触发
  ↓
边界录入和路线判断
  ↓
候选软件挖掘 / 灵魂审查 / 申报场景映射
  ↓
已有软件扫描 / 无软件先做 MVP
  ↓
产品化与 UI 设计门禁
  ↓
权属、AI 辅助、开源依赖和敏感信息审计
  ↓
业务理解和申请口径确认
  ↓
代码候选清单和抽取选择确认
  ↓
申请表辅助字段、操作手册、源代码鉴别材料草稿
  ↓
截图、运行证据和一致性自检
  ↓
Word 三件套生成、回读/结构/版式 QA
  ↓
正式资料和交付自检记录
```

## 与专利 skill 的差异

| 维度 | 专利挖掘 | 软著申请 |
|------|----------|----------|
| 主要审查点 | 技术问题、区别特征、技术效果、授权可能性 | 软件表达、独立开发、权属、鉴别材料和材料一致性 |
| 外部检索 | 查新是主流程 | 外部调研只辅助行业表达和竞品理解 |
| 产出核心 | 技术交底书、图示、查新记录 | 申请表 Word、源程序 Word、操作说明书 Word、截图和 QA 记录 |
| AI 风险 | 夸大技术效果、伪造创新 | 伪造源码、低质生成软件、权属不清、文档空泛 |

## 最低可申请软件门槛

当用户没有软件时，先定义一个窄而真实的 MVP。最低门槛不是“代码行数够”，而是：

- 有明确使用者、业务对象和操作流程。
- 可在本地或浏览器运行。
- 有不少于 3 个真实功能模块或一个完整业务闭环。
- 能说明“谁在什么时间、什么地点、为了哪个申报/审查/归档任务使用”。
- 能映射到至少一种工程实践价值：数据质检、格式标准化、规则计算、异常定位、报告/申报包生成、日志留痕或人工复核。
- 有真实源码结构、版本记录、README、运行方式和测试/验证记录。
- 有可截图的关键状态：初始、配置/输入、运行/处理、结果/导出。
- UI 不应只是裸控件堆叠；应有清晰信息架构、状态反馈、错误提示、结果区和导出/日志入口。
- 说明书能从软件界面和源码证据中写出来。

## 候选挖掘与产品化原则

- 先做候选矩阵，再选择软件方向。候选必须包含业务痛点、工程实践意义、申报联系、现有证据和补强方案。
- 如果现有成果只是脚本、低保真 demo 或简陋界面，先补成真实业务闭环，而不是靠文案包装。
- UI 设计要服务软著材料：关键截图应能直接支撑申请表主要功能和操作说明书章节。
- 对工程工具类软件，界面应偏工作台风格：导航、数据输入、处理状态、结果预览、异常清单、导出和日志清楚可见。
- 不造轮子：基础 UI 组件、图表、地图、文档/表格处理、坐标/GIS/CAD 等能力优先使用成熟库。

## Word 交付原则

- Markdown 是过程稿，不是最终交付。
- 最终资料必须落成 Word 三件套，并保留生成报告。
- 申请表必须按模板标签和合并单元格结构定位，保存后回读关键字段。
- 源代码材料必须来自真实源码，不为凑页数伪造代码。
- 操作说明书必须图文结合，截图来自真实软件运行状态。
- 操作说明书不得包含草稿摘要、过程记录、Markdown 标题符号或 Mermaid 原文；流程图代码必须转换为图片。
